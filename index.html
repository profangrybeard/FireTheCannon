Why we are doing this project

This is your first time inside a Blueprint. The point is not to memorize nodes — it's to feel the moment when you, the artist, take ownership of how an actor behaves in the world. Every cannon, every projectile, every target you place will have you answering the question "what happens when…?"

Blueprint is the artist's access point to gameplay. The components panel works like a Maya/Blender outliner. The event graph is just a flowchart. If you can sketch what should happen on a whiteboard, you can wire it here.

Two sessions, twelve phases. Build small, save often, screenshot constantly.

Phase 2 Anatomy of a Blueprint
Read this section before you click anything. Three minutes here saves you an hour of being lost in the editor.

The four kinds of Blueprint you will meet

Class Blueprint — the most common. A reusable Actor (or component, or other type) you can drop into the level dozens of times. BP_Cannon is a Class Blueprint. Edit it once, every cannon in the world updates.

Level Blueprint — one per level. Use it for level-specific scripting (a door opens when this specific switch is hit). Avoid putting reusable logic here — it can't be reused in another level.

Construction Script — runs in the editor whenever you place or modify the actor. Artist gold: parametric placement (e.g., a fence BP that auto-generates 12 posts based on a length variable). Does not run during gameplay.

Event Graph — runs at runtime. Event BeginPlay, input events, custom events, hit events all live here. This is where gameplay happens.

The components panel is your hierarchy. If you've used Maya, Blender, or Photoshop layers, you already understand it. Each component is a piece of the actor. The root component sets the actor's transform; everything else is parented under it.

Phase 5 Enhanced Input — wire the Fire key
Input belongs to the player, not the cannon. The player presses Fire; the player tells the cannons to fire. This separation is one of the most important patterns in Unreal.

Why two assets for one keypress?

The Input Action describes what can happen ("the player wants to fire"). The Input Mapping Context describes how they trigger it ("Left Mouse Button, on this device, in this game state"). Splitting them lets you re-bind keys, support gamepads, and swap entire control schemes (driving vs walking vs menu) without touching gameplay code.

You will see this two-asset pattern again on every Unreal project for the rest of your career. Learn it now.

We're going to own the input mapping. The template's IMC_Default is fine reference code, but it lives outside _project. As the artist owning this game's controls, you should have your own IMC inside your own folder. We'll duplicate the template's IMC, extend it with our Fire action, then swap the BP to use ours.
Why not Space for Fire? The template's IA_Jump already owns Space. Two actions on the same key in the same context = unpredictable behavior. LMB is also more thematic for "fire" and matches every shooter your students have played. Keep Space for Jump, use LMB for Fire.
BP_ThirdPersonCharacter — Event BeginPlay (after the swap):

Event BeginPlay → Get Controller → Cast to PlayerController
  → Get Local Player → Get Subsystem (EnhancedInputLocalPlayerSubsystem)
  → Add Mapping Context
    Mapping Context = IMC_Cannons (was IMC_Default)
    Priority = 0
Movement broke after the swap? You probably saved IMC_Cannons before the duplicate finished pulling in the inherited mappings. Open IMC_Cannons and confirm IA_Move, IA_Look, IA_Jump, and IA_Fire are all listed. If Move and Look are missing, delete IMC_Cannons and redo the Duplicate from IMC_Default.
Fire prints nothing? Three things in order: (1) Is your default pawn actually BP_ThirdPersonCharacter? Open World Settings → GameMode Override → Default Pawn Class. (2) Did you Save IMC_Cannons after adding IA_Fire? Input assets need their own save. (3) Did the Add Mapping Context node actually run? Drop a Print String right after BeginPlay to verify.

Phase 6 First shot — spawn a cannonball
By the end of this phase, clicking LMB sends a sphere flying out of every cannon. That's Monday's payoff.
BP_Cannon — Fire function:

Fire (entry) → Spawn Actor From Class
  Class = BP_Cannonball
  Spawn Transform ← Make Transform
    Location ← Muzzle → Get World Location
    Rotation ← Muzzle → Get World Rotation
  Collision Handling Override = Always Spawn, Ignore Collisions
Why "Always Spawn, Ignore Collisions"? Spawn Actor's default behavior is to refuse to spawn if the new actor would overlap something. Since the cannonball spawns inside the muzzle, that overlap check would fail every time. We override it.
BP_ThirdPersonCharacter — IA_Fire Triggered:

IA_Fire (Triggered) → Get All Actors of Class
  Actor Class = BP_Cannon
→ For Each Loop
  → (Loop Body) Array Element → Fire (call Fire on each cannon)
Get All Actors of Class is expensive. It scans the entire world. Fine for one shot per second on a level with two cannons. Don't use it on Tick. In a real game you'd cache cannon references in an array on BeginPlay and iterate that — same For Each, no world scan. We're keeping the GAAOC version here because it makes the lesson visible; just remember the better pattern exists for when you do this for real.

Wednesday — Session 2: Impact, Animation, Destruction, Polish
Phase 7 Make the cannonball move and hit
A static sphere is not a projectile. Now we make it physics-believable and reactive.
Projectile Movement vs Simulate Physics. They are not the same thing. ProjectileMovement is a kinematic mover with built-in arc, homing, and bounce options — predictable and cheap. Simulate Physics is full rigid-body simulation — flexible but expensive and unpredictable. For projectiles that just need to fly straight and hit something, ProjectileMovement is the right tool every single time.
BP_Cannonball — On Component Hit:

On Component Hit (Ball) → Apply Damage
  Damaged Actor = Other Actor (from hit event)
  Base Damage = 25.0
  Damage Causer = Self
→ Destroy Actor (target: Self)
Cannonball flies through walls? Open the wall mesh's collision settings → make sure it's not set to NoCollision. If you imported a FAB mesh that has no collision, double-click it → Collision menu → Add Box Simplified Collision (or Auto Convex Collision for complex shapes).
Phase 8 BP_Target — destroyable barrels and posts
Apply Damage on its own does nothing — the actor receiving the damage has to listen for it and decide what to do. That's BP_Target's job.
BP_Target — Event Any Damage:

Event Any Damage → Set Health
  Health = Health - Damage
→ Branch (Condition: Health <= 0)
  True → Spawn Emitter at Location (or Spawn System at Location for Niagara — pick any explosion-y FX from Starter Content)
    Location = Get Actor Location
  → Destroy Actor (Self)

Phase 9 Recoil animation with a Timeline
Timelines are the most artist-native node in Blueprint. If you've ever set keyframes in Maya or scrubbed a curve in After Effects, you already know how to use them.
BP_Cannon — Fire flow with recoil:

Fire (entry) → RecoilTimeline (Play from Start)
  → Spawn Actor From Class (BP_Cannonball — same as before)

RecoilTimeline (Update) → CannonMesh → Set Relative Location
  New Location ← Make Vector
    X = RecoilOffset (from timeline)
    Y = 0, Z = 0
Functions can't contain Timelines. If your Fire is a Function (it is, from Phase 6), you'll get a compile error. Convert Fire from a Function to a Custom Event: My Blueprint → right-click Fire → Convert to Event. Custom Events live in the Event Graph and can host Timelines, Delays, and other latent nodes. Functions can't.

Phase 10 Chaos Destruction — what it is, what it isn't
The headline feature. Used carefully it's the most satisfying thing in this whole project. Used carelessly it's a 30 FPS performance disaster. Read the concept box first.

Chaos Destruction — the honest read

What Chaos is designed for: art-driven, set-piece destruction. A glass window shattering when you walk through. A pillar collapsing in a cinematic. A wooden crate exploding when shot. Hero-moment, on-camera, "look at this cool break" — that's its lane and it's superb at it.

What Chaos is NOT: a true real-time destruction engine. It is not Red Faction Guerrilla's GeoMod. It will not give you a fully-destructible city block where every building piece persists, collides with AI, and survives a multiplayer rollback. Pieces fall asleep, can't be saved across levels easily, are hard to network deterministically, and consume serious CPU when many sim at once.

Mental model: Chaos is a visual effect that happens to use physics, not a physics simulation that happens to look pretty. Treat broken pieces as cosmetic debris — destroy them after a few seconds, don't build gameplay on them.

Field Systems — the real Chaos workflow

Chaos breaks happen because something applies strain to chunks. A cannonball collision can apply strain directly via impulse, but the workflow Epic intends — and what production projects use — is a Field System.

A Field System is a Blueprint-style graph that describes a function over space: "at this position, deliver this much strain (or force, or velocity, or angular impulse) to anything listening." When a Field System Actor is in the level, the Chaos solver evaluates that function every tick and applies the result to nearby Geometry Collections.

Why this matters: Field Systems give you spatial control (radial falloff, planes, boxes), per-chunk targeting (only chunks at this depth, only chunks of this size), and composability (combine multiple fields with operators). It's how Epic does cinematic destruction in their demos. Direct impulse works for one-shot prototypes; Field Systems are how you build a real destruction system.

FS_CannonImpact — Construction Script:

Construction Script → Add Field Command
  Target = External Cluster Strain
  Field ← Radial Falloff
    Magnitude = 1500.0
    Min Range = 0.0, Max Range = 1.0
    Default = 0.0
    Radius = 300.0 (impact area in cm)
    Position = GetActorLocation (self)
    Falloff = Linear
Reading this field in plain English: "At my actor's location, deliver up to 1500 units of External Cluster Strain to any chunks within 300 cm, falling off linearly to zero at the edge." That's enough to break a cluster (threshold 500) but the falloff means only chunks near the impact actually fly.
Why Set Life Span and not Destroy Actor immediately? The Field System Component needs at least one tick to send its commands to the Chaos solver. Destroying on BeginPlay can race the solver and produce no break. 0.1 seconds is generous and still invisible to the player.

BP_Cannonball — On Component Hit (full updated chain):

On Component Hit (Ball) → Spawn Actor From Class
  Class = BP_ImpactField
  Spawn Transform ← Make Transform
    Location = Hit Location (from hit event)
  Collision Handling Override = Always Spawn, Ignore Collisions
→ Apply Damage (existing — for BP_Target)
  Damaged Actor = Other Actor
  Base Damage = 25.0
  Damage Causer = Self
→ Destroy Actor (Self)
Why spawn the field on every hit, not just GC hits? Geometry Collections are the only actors that listen to External Cluster Strain. Walls, targets, and everything else simply ignore the field — no cast required, no branch, no overhead. The field broadcasts; only listeners react. This is one of the most elegant patterns in Chaos.
Performance discipline (non-negotiable): Each BP_ImpactField spawn = one field command in the Chaos solver. At our cannon firing rate this is fine. If you ever loop fire (Bonus B's explosive shells, for instance), watch the frame time — each shell spawns its own field. Limit total simulating chunks across the level to ~150 max. Use the GC's Sleep Threshold (Details → Physics) so resting chunks don't keep ticking. Add a Lifespan override to broken pieces (in the GC's Sim → Cluster Group settings) to despawn them after 5 seconds.
Going further: The Field System editor supports composition — Add Field Command nodes can chain, and operator nodes (Sum, Cull, Conversion) let you combine fields. Try adding a second Add Field Command targeting Linear Velocity with a smaller Radial Falloff to add an outward push to broken chunks, on top of the strain that breaks them. Epic's Content Examples sample project has a "Chaos" map with eight or so pre-built Field System examples worth dissecting after you submit.

Phase 12 Package, capture, submit
Three deliverables go to two different places. Read carefully — this section is where students lose easy points.
Build fails? 90% of first-time package failures are: (1) missing platform SDK — Edit → Editor Preferences → Platforms → Windows → install if prompted; (2) a level reference that's still pointing at the template's default map; (3) a node that compiles in PIE but not in a packaged build (rare at our scope). The Output Log says exactly which. Search the error verbatim before asking.
Comment boxes earn rubric points. In any Event Graph, select a group of nodes → press C → type a label → set the comment color via the comment's Details panel. They show your intent to a grader who has to read your wiring. Use them on every Fire event, every On Hit, every BeginPlay flow.

Bonus A Charge-shot mechanic — hold LMB to power up ADVANCED
Hold the fire button to charge; release to fire. Charge level scales projectile speed. Teaches Enhanced Input triggers, latent timelines, and passing parameters to your Fire event.
BP_ThirdPersonCharacter — IA_Fire (charge mechanic):

IA_Fire (Started) → Set ChargeAmount = 0.0

IA_Fire (Triggered) → Set ChargeAmount
  Value = Clamp (ChargeAmount + DeltaSeconds / MaxChargeTime, 0, 1)
(use Get World Delta Seconds; this normalizes charge to 0–1 over MaxChargeTime)

IA_Fire (Completed) → Get All Actors of Class (BP_Cannon)
→ For Each Loop → Array Element → Fire
  ChargePower = ChargeAmount (new input — see next step)
→ Set ChargeAmount = 0.0 (reset for next shot)
BP_Cannon — Fire (charge integration):

Spawn Actor From Class (BP_Cannonball) → Return Value → Get Projectile Movement Component
→ Set Velocity in Local Space
  New Velocity = Make Vector
    X = Lerp (1500.0, 5000.0, ChargePower)
    Y = 0, Z = 0
Why Set Velocity in Local Space? The cannonball spawns oriented along the muzzle's rotation, so its local +X is "forward out of the cannon." Setting velocity in local space lets you specify "forward" without doing rotation math.
BP_Cannonball's ProjectileMovement has Initial Speed = 3000. Setting Velocity overrides Initial Speed only if Set Velocity runs after BeginPlay (which Spawn Actor → Set Velocity guarantees). If your shots all fly at 3000 regardless of charge, you're hitting the wrong component or your Lerp is upstream of the Spawn.

Bonus B Inheritance — explosive ammo via child Blueprints ADVANCED
Two ammo types — standard and explosive. Press Tab to switch. Teaches the single most important OOP concept you will use in Unreal: child Blueprints inheriting and overriding parent behavior.

Why inheritance instead of "just copy BP_Cannonball"?

If you duplicated BP_Cannonball to make an explosive variant, you'd have two separate Blueprints. Fix a bug in one — you have to remember to fix it in the other. Add a feature to the base — you'd have to copy it across. That's how technical debt is born.

With a child Blueprint, the child inherits the parent's components, variables, and event graph. You only override what's different. Bug fix in the parent → free for the child. New feature in the parent → free for the child.

This is why class hierarchies exist in code. You're about to use it visually, with no syntax.

BP_Cannonball_Explosive — On Component Hit (override):

On Component Hit (Ball) → Spawn Actor From Class (BP_ImpactField)
  Location = Hit Location
→ Apply Radial Damage With Falloff
  Origin = Hit Location
  Damage Inner Radius = 200.0 (full damage zone)
  Damage Outer Radius = 600.0 (falloff zone)
  Base Damage = 75.0
  Minimum Damage = 10.0
  Damage Falloff = 1.0 (linear)
  Damage Type Class = DamageType (default is fine)
  Damage Causer = Self
→ Spawn Emitter at Location (big explosion VFX)
→ Destroy Actor (Self)
Apply Radial Damage With Falloff automatically calls Event Any Damage on every actor inside the outer radius. Your existing BP_Target already listens for that — so explosive shells will damage every target in the blast radius without any further wiring. That's the elegance of working through Apply Damage.
BP_ThirdPersonCharacter — IA_SwitchAmmo (toggle):

IA_SwitchAmmo (Triggered) → Branch (Condition: CurrentAmmoClass == BP_Cannonball)
  True → Set CurrentAmmoClass = BP_Cannonball_Explosive
  False → Set CurrentAmmoClass = BP_Cannonball
→ Print String (text: "Ammo: " + CurrentAmmoClass.GetName())
Going further: Make this an Enum instead of a Class Reference. Define E_AmmoType with values Standard / Explosive / Cluster, give the player a Map (Enum → Class) variable, and switch types via array index instead of a single binary toggle. That's how a real ammo wheel works.
Bonus C Score HUD with UMG ADVANCED
A score that ticks up as targets die, displayed in a Widget Blueprint added to the viewport. Teaches GameMode as the gameplay scoreboard, UMG widget creation, and binding text to a remote variable.
Forgot to set Default Pawn Class? You'll spawn into the level as a floating camera. Open BP_CannonsGameMode → Class Defaults panel → look for the four pawn-class slots (Pawn, Player Controller, HUD, Game State).
WBP_HUD — Get_ScoreText_Text_0 (binding):

Function entry → Get Game Mode → Cast to BP_CannonsGameMode
  Success → As BP_CannonsGameMode → Score
  → Format Text ("Score: {0}", arg = Score)
  → Return Node (Return Value pin)
  Cast Failed → Return Node (Return Value = "Score: 0")
Bindings tick every frame. Cheap for a single Cast and an int read. For anything more expensive (Get All Actors of Class, line traces), don't use a binding — use Event Dispatchers from the Game Mode to push updates only when score changes. We're keeping it simple here.

Bonus D Slow-mo cinematic on Chaos break ADVANCED
When a Geometry Collection breaks, time briefly dilates so the player can savor the shatter. Teaches Set Global Time Dilation, latent Delay nodes, and why "feel" usually comes from time, not visuals.
BP_Cannonball — OnComponentHit (slow-mo branch):

(after BP_ImpactField spawn)
Cast to GeometryCollectionActor (target: Other Actor)
  Success → Set Global Time Dilation (World Context: Self, Time Dilation: 0.25)
  → Delay (Duration: 0.4 — runs in dilated time, so feels like ~1.6s real)
  → Set Global Time Dilation (Time Dilation: 1.0 — restore)
Why 0.25 dilation for 0.4s? 0.25 is "obviously slow but readable." 0.4s of dilated time at 0.25× scale = roughly 1.6 seconds real-world — long enough to register, short enough not to feel sluggish. Tune to taste.
The Delay is a latent node — it can't live in a Function. Your OnComponentHit is an event in the Event Graph, so this works. If you tried to put this in a Function, the Delay node would refuse to compile. (Same restriction as Timelines from Phase 9.)
Multiple breaks during dilation? If two cannonballs hit two GCs within the slow-mo window, the second hit's "Set Time Dilation 1.0" runs first when its Delay completes, and the first hit's "Set 1.0" then runs against an already-normal world. Players won't notice. If they do, gate this with a bool: IsInSlowMo — only enter slow-mo if it's false. Set true on entry, false on exit.
