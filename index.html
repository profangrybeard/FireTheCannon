Why we are doing this project

This is your first time inside a Blueprint. The point is not to memorize nodes — it's to feel the moment when you, the artist, take ownership of how an actor behaves in the world. Every cannon, every projectile, every target you place will have you answering the question "what happens when…?"

Blueprint is the artist's access point to gameplay. The components panel works like a Maya/Blender outliner. The event graph is just a flowchart. If you can sketch what should happen on a whiteboard, you can wire it here.

Two sessions, twelve phases. Build small, save often, screenshot constantly.

Phase 2 Anatomy of a Blueprint
Read this section before you click anything. Three minutes here saves you an hour of being lost in the editor.

The four kinds of Blueprint you will meet

Class Blueprint — the most common. A reusable Actor (or component, or other type) you can drop into the level dozens of times. BP_Cannon is a Class Blueprint. Edit it once, every cannon in the world updates.

Level Blueprint — one per level. Use it for level-specific scripting (a door opens when this specific switch is hit). Avoid putting reusable logic here — it can't be reused in another level.

Construction Script — runs in the editor whenever you place or modify the actor. Artist gold: parametric placement (e.g., a fence BP that auto-generates 12 posts based on a length variable). Does not run during gameplay.

Event Graph — runs at runtime. Event BeginPlay, input events, custom events, hit events all live here. This is where gameplay happens.

The components panel is your hierarchy. If you've used Maya, Blender, or Photoshop layers, you already understand it. Each component is a piece of the actor. The root component sets the actor's transform; everything else is parented under it.

Phase 5 Enhanced Input — wire the Fire key
Input belongs to the player, not the cannon. The player presses Fire; the player tells the cannons to fire. This separation is one of the most important patterns in Unreal.

Why two assets for one keypress?

The Input Action describes what can happen ("the player wants to fire"). The Input Mapping Context describes how they trigger it ("space bar, on this device, in this game state"). Splitting them lets you re-bind keys, support gamepads, and swap entire control schemes (driving vs walking vs menu) without touching gameplay code.

BP_ThirdPersonCharacter — Event Graph (BeginPlay extension):

Event BeginPlay → Get Controller → Cast to PlayerController
  → Get Local Player → Get Subsystem (class: EnhancedInputLocalPlayerSubsystem)
  → Add Mapping Context
    Mapping Context = IMC_Cannons
    Priority = 0
The template already adds its own IMC. Don't replace that chain — branch off it or chain onto its end. Both contexts can be active at once; that's the point of priority. Your IMC_Cannons at priority 0 sits alongside the template's movement IMC.
Nothing happens? Check three things in order: (1) Is your character actually BP_ThirdPersonCharacter when you press play? Open World Settings → confirm GameMode → confirm Default Pawn Class. (2) Did you Save the IMC asset itself, not just the BP? Input assets need their own save. (3) Did the Add Mapping Context node actually run? Drop a Print String right after BeginPlay to verify.

Phase 6 First shot — spawn a cannonball
By the end of this phase, pressing space sends a sphere flying out of every cannon. That's Monday's payoff.
BP_Cannon — Fire function:

Fire (entry) → Spawn Actor From Class
  Class = BP_Cannonball
  Spawn Transform ← Make Transform
    Location ← Muzzle → Get World Location
    Rotation ← Muzzle → Get World Rotation
  Collision Handling Override = Always Spawn, Ignore Collisions
Why "Always Spawn, Ignore Collisions"? Spawn Actor's default behavior is to refuse to spawn if the new actor would overlap something. Since the cannonball spawns inside the muzzle, that overlap check would fail every time. We override it.
BP_ThirdPersonCharacter — IA_Fire Triggered:

IA_Fire (Triggered) → Get All Actors of Class
  Actor Class = BP_Cannon
→ For Each Loop
  → (Loop Body) Array Element → Fire (call Fire on each cannon)
Get All Actors of Class is expensive. It scans the entire world. Fine for one shot per second on a level with two cannons. Don't use it on Tick. In a real game you'd cache references at BeginPlay. We'll talk about that on Wednesday — for now, simplicity wins.

Wednesday — Session 2: Impact, Animation, Destruction, Polish
Phase 7 Make the cannonball move and hit
A static sphere is not a projectile. Now we make it physics-believable and reactive.
Projectile Movement vs Simulate Physics. They are not the same thing. ProjectileMovement is a kinematic mover with built-in arc, homing, and bounce options — predictable and cheap. Simulate Physics is full rigid-body simulation — flexible but expensive and unpredictable. For projectiles that just need to fly straight and hit something, ProjectileMovement is the right tool every single time.
BP_Cannonball — On Component Hit:

On Component Hit (Ball) → Apply Damage
  Damaged Actor = Other Actor (from hit event)
  Base Damage = 25.0
  Damage Causer = Self
→ Destroy Actor (target: Self)
Cannonball flies through walls? Open the wall mesh's collision settings → make sure it's not set to NoCollision. If you imported a FAB mesh that has no collision, double-click it → Collision menu → Add Box Simplified Collision (or Auto Convex Collision for complex shapes).
Phase 8 BP_Target — destroyable barrels and posts
Apply Damage on its own does nothing — the actor receiving the damage has to listen for it and decide what to do. That's BP_Target's job.
BP_Target — Event Any Damage:

Event Any Damage → Set Health
  Health = Health - Damage
→ Branch (Condition: Health <= 0)
  True → Spawn Emitter at Location (or Spawn System at Location for Niagara — pick any explosion-y FX from Starter Content)
    Location = Get Actor Location
  → Destroy Actor (Self)

Phase 9 Recoil animation with a Timeline
Timelines are the most artist-native node in Blueprint. If you've ever set keyframes in Maya or scrubbed a curve in After Effects, you already know how to use them.
BP_Cannon — Fire flow with recoil:

Fire (entry) → RecoilTimeline (Play from Start)
  → Spawn Actor From Class (BP_Cannonball — same as before)

RecoilTimeline (Update) → CannonMesh → Set Relative Location
  New Location ← Make Vector
    X = RecoilOffset (from timeline)
    Y = 0, Z = 0
Functions can't contain Timelines. If your Fire is a Function (it is, from Phase 6), you'll get a compile error. Convert Fire from a Function to a Custom Event: My Blueprint → right-click Fire → Convert to Event. Custom Events live in the Event Graph and can host Timelines, Delays, and other latent nodes. Functions can't.

Phase 10 Chaos Destruction — what it is, what it isn't
The headline feature. Used carefully it's the most satisfying thing in this whole project. Used carelessly it's a 30 FPS performance disaster. Read the concept box first.

Chaos Destruction — the honest read

What Chaos is designed for: art-driven, set-piece destruction. A glass window shattering when you walk through. A pillar collapsing in a cinematic. A wooden crate exploding when shot. Hero-moment, on-camera, "look at this cool break" — that's its lane and it's superb at it.

What Chaos is NOT: a true real-time destruction engine. It is not Red Faction Guerrilla's GeoMod. It will not give you a fully-destructible city block where every building piece persists, collides with AI, and survives a multiplayer rollback. Pieces fall asleep, can't be saved across levels easily, are hard to network deterministically, and consume serious CPU when many sim at once.

Mental model: Chaos is a visual effect that happens to use physics, not a physics simulation that happens to look pretty. Treat broken pieces as cosmetic debris — destroy them after a few seconds, don't build gameplay on them.

BP_Cannonball — extend On Component Hit:

On Component Hit → Apply Damage (existing)
→ Cast to GeometryCollectionActor (target: Other Actor)
  Success → As GeometryCollectionActor → Get Geometry Collection Component
  → Add Impulse At Location
    Impulse = Get Velocity (self) * 50
    Location = Hit Location (from hit event)
→ Destroy Actor (Self)
Performance discipline: Keep total simulating chunks under ~150 across the level at any time. Use the Geometry Collection's Sleep settings to put debris to sleep after motion stops. Add a Lifespan override on broken pieces to despawn them after 5 seconds. This is non-negotiable — without it your framerate will tank as students stack up debris.
Field Systems vs impulses. The "Field System" approach (Master Field → Radial Falloff Field → Strain) is more flexible and the "real" Chaos workflow. We're using direct impulse here for simplicity and because it ties cleanly to the physics of a cannonball hit. If you want to go deeper, the Epic Sample "Content Examples" project has a Chaos map with Field examples — open it after you submit.

Phase 12 Package, capture, submit
Three deliverables go to two different places. Read carefully — this section is where students lose easy points.
Build fails? 90% of first-time package failures are: (1) missing platform SDK — Edit → Editor Preferences → Platforms → Windows → install if prompted; (2) a level reference that's still pointing at the template's default map; (3) a node that compiles in PIE but not in a packaged build (rare at our scope). The Output Log says exactly which. Search the error verbatim before asking.
Comment boxes earn rubric points. In any Event Graph, select a group of nodes → press C → type a label → set the comment color via the comment's Details panel. They show your intent to a grader who has to read your wiring. Use them on every Fire event, every On Hit, every BeginPlay flow.
